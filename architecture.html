Architecture Overview

k3c — local k3s clusters on Apple container

A single Go binary that creates and operates k3s clusters and a Docker sidecar as lightweight VMs on Apple's Virtualization.framework — with corporate-network egress, snapshots, and memory management handled on the host.

The engine never links the runtime: it drives the container CLI as a subprocess (resolved by runtime/, which extracts the embedded fork to ~/.cache/k3c/runtime/<ver>). The gvnet netstack likewise runs out-of-process — one detached gvnet per VM — keeping gvisor out of the main binary.

3Host daemons

One detached k3c daemons run process owns every host listener. Ports bind 0.0.0.0 but are source-restricted to loopback + vmnet; contested ports (e.g. :443) are arbitrated — spliced to the docker sidecar when it is the active target.

Non-listener loops in the same process: auto-reclaim (balloon memory back every 10 min), CPU-priority reconcile, pull-cache prune, sidecar published-port mirroring, and the stable docker.sock engine socket with per-connection routing (§4).

4Key data paths

Egress (transparent gvnet — the default)

Every VM gets a second NIC on a private gvnet /24; the entrypoint repoints the default route and DNS at it. The gvisor netstack terminates guest TCP/IP and re-originates each connection from host sockets — so VPN/Zscaler, proxies, and corporate DNS are handled host-side with zero per-domain config. The legacy mode (egress.transparent: false) instead combines the CONNECT proxy, the SNI gateway, and a CoreDNS override that resolves egress domains to the vmnet gateway.

Image pulls

Guests pull through the pull-cache (192.168.64.1:5011) as the first mirror for every registry; misses fall through to real upstreams via the proxy. Snapshot image closures are pinned in the cache so frozen snapshots can rehydrate.

Docker engine socket (sidecar)

The host docker.sock routes each connection by its first HTTP request: hijacked streams (exec/attach) go over the full-duplex --publish-socket vsock bridge (Apple's loopback publish drops hijacks); everything else goes to the Apple-published loopback 127.0.0.1:2375, which — unlike the bridge — doesn't head-of-line-block an undrained stream (the Testcontainers wait.ForLog fix). The loopback route is gated on a cached /_ping health probe, because a boot-restored runtime process can lose its macOS Local Network grant and leave the publish listening-but-dead.

Snapshots

warm

APFS clone of rootfs + VM RAM image + machine identity. Resumes in place, instantly. Restore reclaims the snapshot-time IP (desired-address re-allocation).

cold

Full rootfs clone, no machine state. Boots fresh in ~30–60 s; IP-independent; less RAM held after restore.

frozen

Logical extract: datastore + PVC data + image manifest. Smallest; thaws in minutes, rehydrating images from the pull-cache.

6Release pipeline

Tag v* → goreleaser: clones the pinned philipparndt/container + containerization forks, builds the Swift runtime, stages it with the cross-compiled helpers (gvnet, k3c-docker-fwd, 16K kernel) into runtime/payload/container-runtime.tar.gz, then builds the Go binary with -tags bundled (go:embed). Publishes the GitHub release + Homebrew formula (philipparndt/k3c/k3c). darwin/arm64 only. Fork refs are pinned commits — bumped deliberately, never branch tips.

7Drift & improvement — where fast growth is showing

Findings from a structural review + a docs-vs-code audit (July 2026), ranked by how soon they bite. Full details in the accompanying review.

Fix docsEgress default is inverted in every prose doc

Code defaults transparent gvnet egress ON; ARCHITECTURE.md, README, and the host-egress spec all describe the CONNECT-proxy/SNI path as the default and transparent as opt-in. New users will configure the legacy mode believing it's the normal one.

DOOne doc pass: flip the framing, document egress.transparent, fix the README startup sample.

RefactorCluster vs sidecar is a copy-paste fork

Snapshot save/restore, lifecycle (up/down/pause/resume/suspend/reclaim), and the CLI trees exist twice — near-identical, already diverging (meta filename, registry handling), and every fix must land twice. This is the biggest driver of future bugs.

DOOne snapshot/lifecycle engine behind a small "snapshottable VM" interface; parameterize the CLI tree.

RefactorBusiness logic rides on fragile CLI text parsing

~23 sites column-index-parse container stdout (strings.Fields + positional indexing, some unbounded [0]). Apple's CLI is young; any output change breaks silently.

DOA typed container client using --format json everywhere; it doubles as the test seam.

TestCoverage is inverted — riskiest code untested

22 of 29 cluster/ source files have no test, including snapshot restore, daemons, and the whole docker lifecycle — exactly the most-changed areas. cmd/ (67 commands) has zero tests.

DOAfter the client seam: table tests for both restore paths against a fake client.

Fix docsShipped features without spec or README coverage

Frozen snapshots, export --slim/--thin, k3c web, memory knobs (autoReclaim, memoryPolicy, memoryHeadroom), cluster repair/import-run, info — all live in code, none in specs/README. Five unarchived openspec changes hold the missing specs.

DOArchive the five pending openspec changes into specs; extend the README quick-ref.

RefactorConfig: two ~50-field structs + scattered state files

FileConfig (53 fields) and Config (48) are merged by hand — a new field touches five places. Runtime state (active.yaml, ports.yaml, daemons.version) is read/written ad-hoc from many files (71 raw file ops in cluster/).

DODerive Config from FileConfig; one typed state package owning the on-disk files.

HardenSleep-based sequencing & best-effort error swallowing

17 non-test time.Sleep settle-waits; the code documents real stop/start races with the runtime. 134 ignored-error sites concentrate in snapshot/daemon/restore paths, where "best-effort" recovery can make failure look like success.

DOWait-for-state helpers with timeouts; triage ignored errors on save/restore paths to at least log-and-surface.

Suggested order: 1) the two doc passes (cheap, stops onboarding damage) · 2) typed container client (unlocks testing) · 3) de-fork snapshot/lifecycle · 4) config/state consolidation · then split cluster/ along the §2 cohesion table, pull-cache first (least coupled).
